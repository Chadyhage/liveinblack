01
Vision produit & contexte business

Concept

LIVEINBLACK est une marketplace événementielle dédiée au nightlife, principalement ciblant la diaspora africaine en France et les marchés d'Afrique de l'Ouest. La plateforme met en relation trois types d'acteurs :

ActeurRôleValeur apportée
ClientsAcheteurs de billets, spectateursDécouvrir & réserver des événements nightlife
OrganisateursCréateurs d'événements (boîtes de nuit, associations...)Vendre des billets, gérer la jauge, booster la visibilité
PrestatairesDJs, sono, sécurité, décoration...Proposer leurs services aux organisateurs

Marchés cibles

La plateforme supporte actuellement 5 régions avec détection automatique par géolocalisation :

🇫🇷 France  ·  🇨🇮 Côte d'Ivoire  ·  🇬🇭 Ghana  ·  🇹🇬 Togo  ·  🇧🇯 Bénin

Modèle économique

Vente de billets

Commission sur chaque billet vendu via Stripe. Les organisateurs créent des "places" (catégories) avec prix et capacité indépendants.

Système de Boost

Les organisateurs paient pour positionner leur événement en Top 1, Top 2 ou Top 3 sur la HomePage. Payant, par durée et par région.

Services prestataires

Mise en relation rémunérée entre organisateurs et prestataires (DJs, équipement, venues).

Événements privés

Événements accessibles uniquement via un code secret partagé. Niche premium possible.

⚠️
Statut légal en cours : La structure juridique (SIREN, RCS, capital) n'est pas encore enregistrée au moment de la rédaction. Les pages légales contiennent des placeholders dans src/data/legal.js. À compléter dès immatriculation.

Architecture des données & Firestore

Principe : Write-through cache

💡
Pattern central de l'app : toutes les données sont lues depuis localStorage (préfixe lib_) pour des performances instantanées, puis synchronisées en arrière-plan avec Firestore pour la persistance cross-device. Ce pattern évite les états de chargement pour l'utilisateur.
Login → syncOnLogin(uid)
Charge toutes les données Firestore de l'utilisateur dans localStorage. Les listeners temps réel sont attachés.
Lecture → localStorage d'abord
Les composants lisent localStorage directement via les utils. Réponse instantanée, pas de spinner.
Écriture → localStorage + syncDoc() fire-and-forget
L'UI se met à jour immédiatement. Firestore est mis à jour en background sans bloquer.
Retour dans l'onglet → resync automatique
Un listener visibilitychange déclenche une resync depuis Firestore quand l'onglet redevient actif.

Fonctions utilitaires Firestore (utils/firestore-sync.js)

FonctionDescription
syncOnLogin(uid)Sync complète au login — appeler une seule fois
syncDoc(path, data)Écriture fire-and-forget (non bloquant)
syncDocAwaitable(path, data)Écriture avec await (utiliser pour données critiques)
loadDoc(path)Lecture async depuis Firestore
listenEvents(callback)Listener temps réel sur tous les événements
listenDoc(path, callback)Listener temps réel sur un document

⚠️
Import dynamique obligatoire : Ne jamais importer firestore-sync en static dans les pages. Toujours utiliser import('../utils/firestore-sync').then(({ syncDoc }) => ...) pour éviter les problèmes de chargement.

Rôles & permissions

RôleStatuts possiblesNavigationAccès spéciaux

Flux d'onboarding organisateur / prestataire

Inscription avec rôle org/prest
status = "onboarding", compte créé dans Firestore
OnboardingGuard intercepte
Redirige vers /onboarding-organisateur ou /onboarding-prestataire si status ≠ active
Soumission du dossier
status = "pending", dossier visible dans AgentPage pour l'admin
Validation admin (AgentPage)
status = "active" → accès complet. status = "rejected" → message d'erreur

Pages & composants

Pages critiques

LoginPage.jsx (~1424 lignes) — Page la plus complexe

Gère : connexion email/password, inscription avec rôle, Google/Apple OAuth, sélecteur multi-comptes, validation email, reset password, écran "compte en attente", détection super admin. Ne pas refactoriser sans tests approfondis.

HomePage.jsx (~819 lignes)

Géolocalisation → Nominatim → région → top 3 events filtrés. Contient la section "Comment ça marche" pour visiteurs non connectés, GooeyText animation, globe 3D conditionnel. La logique de filtrage des events (publishAt, closingDate, grace period 48h, boost position) est dans cette page.

EventsPage.jsx (~749 lignes)

Filtres catégorie/recherche, EventCard composant, mode partage ?share= pour envoyer un événement via messagerie, unlock événements privés (code stocké dans lib_event_codes).

AgentPage.jsx — Dashboard admin

Métriques business, graphique de croissance, onglet candidatures organisateurs/prestataires, liste utilisateurs, statistiques globales. Accessible uniquement avec rôle agent.

MessagingPage.jsx

Chat complet avec : conversations directes + groupes, typing indicators temps réel, réactions aux messages (émojis), messages épinglés, suppression (soi / tous), statut lecture/livraison, partage d'événements, polls (types 'poll' ET 'event_poll').

Composants critiques

Layout.jsx

Wrapper global. Contient handleProtectedNav(path) qui intercepte les navigations vers des routes protégées et ouvre AuthModal si l'utilisateur n'est pas connecté. Le bottom nav mobile est en position: fixed (pas sticky).

AuthModal.jsx

Modal global pour les prompts de connexion. Appelé via openAuthModal() depuis useAuth(). Ne pas dupliquer la logique auth ici — LoginPage.jsx est la source de vérité.

EventCard.jsx

Carte événement réutilisable avec variants image/gradient, hover animations, tags catégorie, mode partage. Utilisée sur HomePage et EventsPage.

Points critiques & problèmes connus

Problèmes identifiés (historique git)

ProblèmeStatutDescription
Event cross-device disparu Corrigé Un événement apparaissait puis disparaissait sur d'autres devices. Fix : sync Firestore correctement awaitée.
Sync Firestore non awaitée Corrigé syncDoc() non awaitée sur certaines opérations critiques causait des données perdues. Migré vers syncDocAwaitable().
Bandeau d'erreur invisible Corrigé L'UI d'erreur ne s'affichait pas correctement.
Sync Firestore cross-device À tester La sync bidirectionnelle n'a pas encore été testée en conditions réelles sur plusieurs appareils simultanément.
Webhooks Stripe manquants À faire Confirmation de paiement côté serveur non implémentée. Actuellement, on vérifie la session côté client uniquement (risque faible mais pas robuste).
Super admin hard-codé À améliorer Email hagechady4@gmail.com hard-codé dans LoginPage.jsx. À déplacer dans Firestore.
Données légales incomplètes En attente SIREN, RCS, capital non renseignés dans legal.js. À compléter après immatriculation.

Zones fragiles — à ne pas casser

⛔
handleProtectedNav(path) dans Layout.jsx : intercepte toute navigation protégée. Toute modification peut casser la protection de toutes les routes de l'app.
⛔
voteOnPoll dans messaging.js : gère les types 'poll' ET 'event_poll'. Ne séparer qu'avec des tests complets.
⛔
Bottom nav mobile : doit rester en position: fixed, pas sticky. Changement connu pour casser l'affichage sur certains devices.
⛔
Import dynamique firestore-sync : ne jamais passer en import statique dans les pages. Utiliser import('../utils/firestore-sync').then(...).

Conseils au développeur entrant

👋
Pour commencer : Lire AuthContext.js, firestore-sync.js et Layout.jsx en priorité. Ces 3 fichiers sont la colonne vertébrale de l'app. Tout le reste s'appuie sur ces fondations.
🧪
Pour tester : Créer 3 comptes test (client, organisateur, agent) avec des emails différents. L'organisateur doit passer par le workflow d'approbation admin avant d'avoir accès. Utiliser les clés Stripe test pour les paiements.
🤖
Contexte particulier : Cette application a été entièrement construite en collaboration avec une IA (Claude / Anthropic). Le code est fonctionnel mais certaines parties manquent de commentaires et de tests unitaires. Procéder avec méthode, tester chaque modification, et documenter les changements importants.
